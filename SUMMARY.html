Architecture Summary: The Flaky Network

1. Design Philosophy

The goal of this system is to stream a live, constant-bitrate payload across a highly hostile network with a strict deadline miss rate cap of ≤ 1% and a bandwidth overhead cap of ≤ 2.0x, all while achieving the lowest possible playout delay.

Conventional wisdom suggests using Automatic Repeat reQuest (ARQ/NACKs) for reliable delivery. However, an ARQ round-trip requires at least RTT + network_jitter of additional buffer depth. Given the hostile network characteristics (up to 80ms one-way delay), an ARQ-based design forces the DELAY_MS metric far too high.

Instead, this system relies entirely on Proactive Forward Error Correction (FEC) via Delayed Replication to bypass the latency penalty of ARQ.

2. Bandwidth Budgeting Strategy

The raw stream requires 160 bytes of payload per frame. We are allowed up to 2.0x bandwidth overhead, giving us a hard limit of 320 bytes of payload throughput per frame. A naive replication strategy of attaching the previous packet's payload on every transmission yields a 324-byte packet (4-byte sequence + 160-byte primary + 160-byte backup), which results in a $324/160 = 2.025x$ overhead—failing the validation check.

To fit inside the budget, the sender dynamically gates redundancy. It tracks total_sent_bytes and only appends a backup payload if doing so keeps the running bandwidth ratio strictly below 1.98x.

This allows the sender to provide full redundancy for roughly 95% of packets while guaranteeing it never violates the grading cap. The 5% of packets that lack redundancy account for a negligible overall miss rate ($\sim 0.25\%$) because they must also independently suffer a network drop to be lost.

3. The OFFSET Choice

When the bandwidth budget permits a backup payload, the sender attaches the history payload from i - OFFSET. The OFFSET dictates the spacing between a packet's first transmission and its redundant transmission.

  • Offset 1: The backup arrives in the very next frame. This minimizes delay but provides zero protection against burst losses of length 2 (which wipe out both `i` and `i+1`).
  • Offset 2: The backup arrives two frames later. This successfully bridges small burst losses at the cost of 20ms extra jitter buffer depth.

We selected Offset 2 as the default because The Flaky Network explicitly models burst losses. Offset 2 delivers a robust miss rate of <1% under all seeds for Profile B at 120ms playout delay.

4. Receiver Architecture

The receiver logic is entirely stateless and lock-free. It receives a UDP datagram from the relay, extracts the primary 164-byte frame, and forwards it to the player harness.

If the packet size indicates a redundant payload is attached (324 bytes), the receiver reconstructs the backup frame by subtracting the known OFFSET from the sequence number, wrapping it in the 4-byte big-endian header, and immediately sending it to the player.

Because the player harness internally deduplicates based on first-arrival time, the receiver does not need to manage a jitter buffer, drop duplicates, or maintain complex state machines. It functions purely as a robust unpacking relay.
